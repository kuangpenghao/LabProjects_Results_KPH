# 第006主版本：标签链与 解析图信号审计

## 结论

本主版本合并标签链后处理、上下文细分规则、小规则包、标签分数边界、chart-level 候选和 pre-decode bias 的完整复盘。结论是：局部标签信号存在，但覆盖窄、跨域触发少，且难以稳定进入最终树，因此整体降级为诊断资产。

## 关键证据

- 第一版标签链在 PTB dev 小涨，但 MCTB 预测上零触发，不能进入 MCTB 评分。
- 上下文细分候选在 MCTB 128x5 子集宏 F1 从 0.855764852 到 0.855882776，delta +0.000117925，但只触发 2 次。
- 两规则小包在 MCTB 128x5 子集 macro delta 为 +0.000234509，但五域 640 句只触发 3 棵树。
- chart-level 诊断在完整 PTB dev 中发现 2318 个 top-3 备选标签潜在修正事件，但 MCTB 已解码树触发很低。
- pre-decode label bias 没有把这些局部信号可靠转化成最终树收益。

## 深层判断

局部标签候选不是没有，而是“有候选”和“最终树变好”之间缺少可靠桥梁。继续堆局部规则很容易获得小样本正例，但很难变成稳健跨域提升。

## 路线变化

本版本后，局部标签链和 chart-label 校准不再作为主线扩大。保留为诊断资产，用来解释候选空间；主线转向整树候选池和 selector。

### 阶段 1：标签链后处理与 基线收尾判断

更新时间：2026-04-29 03:17 CST

#### 1. 本轮结论

- 远程全量 Berkeley+BERT 基线 已自动停止并完成最终测试。最佳 dev F1 为 95.52，PTB23 test F1 为 95.60；MCTB_en dialogue/forum/law/literature/review 分别为 85.59/86.36/92.08/85.74/83.81，五域宏平均 86.72。
- baseline 的最后一个 review 域最初中断，不是模型失败，而是远端旧版 parser 对 BERT 空分词缺少 `[UNK]` fallback。已补最小 fallback，只重跑 review 评测，没有重训模型；因此五域结果可以作为全量 baseline 复现表使用。
- 标签链后处理第一版候选在 PTB dev 上是干净的正向小信号：1700 句里只改 13 句、14 个标签，labeled F1 从 94.2777 到 94.3341，边界和 split 完全不变，标签回退为 0；但它在 5 个 MCTB 控制预测子集上零触发，已降级为 PTB 标签习惯校准。
- 上下文细分筛选找到了更值得推进的下一组候选：`ADJP -> VP` with `VBN PP` 加边缘 POS 或父标签条件可以保留 PTB dev 救回并去掉回退，其中 `edges=VBN/NN, len=6-10` 有 3 个 PTB dev label rescue、0 个 regression、2 次 MCTB 控制预测触发。

#### 2. 本轮目标和上下文

本轮从 label-conditioned rule CKY 原型和 Berkeley/BERT 后处理检查接上来。上一轮证明了解码层和规则层可以动，但真实 parser 路径里必须先确保不破坏原预测，再考虑任何跨域评测。

本轮要解决三个具体问题：

1. 第一个真实改变预测的标签链后处理候选是否值得继续放到 MCTB 上评分。
2. 全量 Berkeley+BERT 基线 是否已经平台期、是否应该手动停掉，以及最终五域复现表是否可用。
3. 按用户要求，把迭代报告改成小结论后及时更新和提交，而不是最后一次性补大报告。

比较约束保持不变：训练只用 PTB；MCTB_en 只能用于最终或预测暴露检查，不能用目标域 gold 标签选择规则。

#### 3. 思路进展与依据

这轮最重要的判断不是只看 F1 是否上涨，而是看上涨背后的机制是否能迁移。

标签链候选的 PTB dev 指标很好看，但更深检查发现它只命中了两类很窄的同边界标签链：

- `NP` + `CD CD` 改成 `NP+QP`
- `ADJP` + `VBN PP` 改成 `VP`

这些规则在 PTB 训练树里有明确 gold 习惯，也能解释 PTB dev 的 14 个标签修复；但在 MCTB 控制预测里一次都不触发。因此继续花时间做五域 MCTB 评分没有意义，预计只会得到无改动或近似无改动结果。

后续把候选 signature 拆得更细以后，`ADJP -> VP` 这一类变得更有价值。粗粒度 `ADJP -> VP` 长跨度候选有 PTB dev 救回，但也有 1 个标签回退；加入父标签或左右边缘 POS 后，可以把正负样例分开。当前最强的是 `edges=VBN/NN, len=6-10`：PTB dev 3 次救回、0 次回退，MCTB 控制预测 2 次触发。

#### 4. 具体算法与工程实现

本轮实现和检查了一个保守的 post-decode label-chain calibration：

- 它只改同一括号边界上的标签链，不移动边界。
- 它只允许训练树中高频且方向明确的窄规则。
- 它在输出后处理层运行，不重新训练模型。

同时保留了一个失败探针：duplicate NP unary collapse。该探针会改预测字符串，但 PTB dev aggregate labeled F1 下降，且出现 label regressions，因此不进入下一步。

#### 5. 中间问题与解决方法

第一个问题是 PTB dev 正向信号容易误导。解决方式是补上机制检查：

- 检查 PTB 训练树中对应 child signature 的 gold 标签分布。
- 检查 PTB dev 命中是否是 label rescue，而不是边界或 split 改动带来的连带效果。
- 检查 MCTB 控制预测中是否会真实触发。

第二个问题是 baseline 平台期是否要手动杀掉。检查训练代码后确认，它不是无限训练。默认停止条件约等于最佳 dev 后 18 个 epoch 无提升，随后脚本会自动生成最终 `metrics.tsv` 和 `summary.md`。因此当时最省事且证据最完整的选择是继续等待自动收尾。

第三个问题是 review 域最终评测中断。定位结果是 MCTB review 中存在一个 token 被旧版 BERT tokenizer 分成空列表，远端 source 缺少本地新版里已有的空分词 fallback。修复方式是把空分词映射到 `[UNK]`，只重跑 review 测试；同时用 run-local Python overlay 补齐旧版 Berkeley+BERT 依赖，并通过 Slurm GPU 重新加载同一个 best checkpoint。

#### 6. 实验设计

标签链候选采用 PTB dev 真实预测对照：

- 对照：Berkeley+BERT control prediction。
- 机制：同一预测上应用标签链后处理。
- 指标：labeled F1、unlabeled F1、changed sentences、label rescues/regressions、boundary delta、split delta、rule signature changes。
- 成功标准：不改边界、不增加标签回退、PTB dev 至少不下降，并且 MCTB 控制预测有非零触发。
- 降级条件：MCTB 控制预测零触发，或 PTB dev 出现边界/标签回退风险。

baseline 监控采用远程日志和本地代码双重确认：

- 远程训练最终自动停止，当前最佳 checkpoint 为 dev F1 95.52。
- 本地启动脚本确认训练完成后会自动测试 PTB23 与 5 个 MCTB 领域。
- 训练主程序确认存在基于 dev 无提升的停止条件。
- review 域修复只作用于评测阶段，不改变训练数据、模型权重或前四个已完成测试结果。

#### 8. 评价、经验与教训

这轮把“能不能涨一点”升级成了“涨点是否代表可迁移机制”。标签链候选是一个很好的反例：它在 PTB dev 上干净、可解释、无副作用，但 MCTB 预测零触发，说明它不能解决跨领域表现提升的核心问题。

以后类似后处理候选必须先过三道门：

1. PTB dev 不破坏边界和标签。
2. 规则背后有训练树分布或解析机制解释。
3. 目标域预测上会真实触发，但不能用目标域标签挑规则。

baseline 方面，本轮也确认 3090 的全量训练速度可接受。完整训练约 5 小时自动收尾；其中最终单域 review 评测在 GPU 上约 5 秒完成。除非出现异常、长时间无日志、或最终测试迟迟不生成，否则不用因为平台期就提前杀任务。

2. 优先推进 `ADJP -> VP` 的上下文细分候选，尤其是 `edges=VBN/NN, len=6-10`；在正式五域评分前仍先做 PTB dev 安全检查和 MCTB 预测触发检查。

### 阶段 2：上下文细分标签链候选小规模验证

更新时间：2026-04-29 03:30 CST

#### 1. 本轮结论

- 第 014 轮选出的下一候选 `ADJP -> VP, child TOKEN:VBN PP, len=6-10, edges=VBN/NN` 通过了 PTB dev 安全审计：3 个标签救回、0 个标签回退，边界变化 0，split 变化 0。
- 该候选在 MCTB 预测上不是 no-op：五域 128 句子集里有 2 次真实触发，分别出现在 literature 和 review。
- 小规模 MCTB 子集评分非负：宏 F1 从 0.855764852 到 0.855882776，delta +0.000117925；但触发量只有 2 次，所以这只是候选门槛通过，不是正式性能结论。
- 下一步应该继续搜索同类低风险规则，优先扩大“PTB 安全 + MCTB 非零触发”的候选池，而不是立刻把这一条单独推向全量测试。

#### 2. 本轮目标和上下文

第 014 轮完成了全量 Berkeley+BERT 基线收尾，并确认第一版标签链后处理虽然 PTB dev 小涨，但在 MCTB 预测上零触发，因此不能进入 MCTB 评分。

本轮目标是验证一个更细的上下文规则：只在预测 span 满足 `ADJP -> VP`、子结构为 `TOKEN:VBN PP`、长度为 `6-10`、左右边缘 POS 为 `VBN/NN` 时改标签。这个候选来自 PTB train/dev 与 MCTB 仅预测 exposure 筛选，不使用 MCTB gold 标签做筛选。

比较约束保持不变：

- 训练仍是原始 PTB-only。
- MCTB_en 只作为测试；筛选阶段不读 MCTB 标签。
- 先过 PTB dev 安全审计，再做小规模 MCTB 子集评分。
- 任何子集分数都只能作为候选门槛信号，不是正式全量 baseline 对比。

#### 3. 思路进展与依据

粗粒度 `ADJP -> VP` 在 base/bucket 层面有 4 个救回但也有 1 个回退，不能直接推广。上下文细分后，`edges=VBN/NN` 把同一个方向切得更干净：

- train 支持：593 / 679，占比 0.873。
- PTB dev 筛选：3 hits，3 rescues，0 regressions。
- MCTB 仅预测 exposure：2 hits。

这个模式符合当前 post-decode calibration 线的核心要求：只做同边界标签校准，不改边界、不改 split，不进入 CKY 内部。

#### 4. 具体算法与工程实现

本轮把标签链应用脚本从硬编码规则扩展为可读取候选筛选结果的通用版本。新的应用方式可以直接从 `candidate_screen.json` 选取第 N 名候选，保留候选里的：

- from/to label；
- child signature；
- length key；
- parent label；
- left/right edge POS；
- train/PTB dev/MCTB exposure 元数据。

实现上仍然只做 best-tree 后处理：遍历已解码树，对匹配候选签名的内部节点改 label，保持叶子、跨度、边界和子树结构不变。

#### 5. 中间问题与解决方法

小规模 MCTB 评分汇总时，审计 JSON 对完全未变化的领域不会写 `changed_sentences` 字段。第一次汇总因此触发了一个缺省键错误；审计本身已经完成，修复方法是在 summary 生成时把未变化领域的缺省计数解释为 0。

这不影响评分结果，因为原始审计 JSON 已经写出，后续只是 summary 表格生成逻辑修正。

#### 6. 实验设计

##### PTB dev 安全审计

- gold：true PTB dev。
- control：同一强 checkpoint 的 PTB dev control prediction。
- mechanism：应用候选后的 PTB dev prediction。
- 成功标准：labeled F1 非负，unlabeled F1 不变，边界变化 0，split 变化 0，标签回退 0，且候选必须真实改变预测。

##### MCTB 仅预测触发检查

- 输入：五个 MCTB 领域的 control predictions。
- 不读 MCTB gold 标签，只统计候选是否真实改动预测。
- 成功标准：至少 1 次真实触发。

##### MCTB 小规模子集评分

- 子集：每个 MCTB 领域 128 句，复用同一 strong-checkpoint audit set。
- 评分目的：确认触发不是明显负信号。
- 成功标准：macro labeled F1 非负，且不引入边界或 split 变化。

### 阶段 5：图表级标签备选筛选

#### 结论

本轮完成了图表级标签备选筛选，并把直接后处理标签替换路线降级。完整 PTB dev 1700 句中，模型图表里存在 `2318` 个“金标在 top-3 非首选标签中”的潜在修正事件，说明上一轮 best-tree 小规则覆盖窄不是因为模型完全没有候选信号；但这些信号大多没有进入最终解码树，MCTB 128x5 子集里最高的已解码树触发只有 `4` 次。

这意味着图表层信号可以作为诊断和后续解码前干预线索，但不适合作为新的直接 post-decode 小规则包来放大。

#### 本轮目标

1. 检查 strong Berkeley+BERT checkpoint 的 chart label top-k 中是否存在比 best-tree 标签链更高覆盖的 PTB dev 安全候选。
2. 只用 PTB dev gold 判断 rescue/regression，不使用 MCTB_en gold 做筛选或调参。
3. 对 MCTB 只统计预测句子上的 chart-level / decoded-tree exposure，用来判断候选是否可能真正影响跨域预测。

#### 实现

新增诊断脚本 `screen_chart_label_alternatives.py`。脚本读取同一 strong checkpoint，枚举 chart 中每个 span 的 top-k label；对 PTB dev，记录“当前解码标签错误、top-k 备选标签等于 gold”的 rescue opportunity，同时统计会把正确标签改错的 regression opportunity；对 MCTB 五域，显式启用 no-gold 模式，只统计同类 label-pair 条件在 chart 和已解码树中的出现次数。

筛选条件是宽松诊断门槛，不是直接可发布规则：PTB rescues 至少 1、PTB regressions 为 0、MCTB chart exposure 至少 1。non-gold churn 被保留为风险指标，因为它可能表示同一边界上的错误标签互换。

#### 实验过程

- 远端 CPU 2 句 smoke 通过，证明 checkpoint 读取、chart 分数导出、no-gold MCTB 模式和报告写出都能工作。
- 128 句级别检查显示方向有信号：PTB dev 128 句有 `294` 个潜在修正事件、`154` 个宽松通过候选。
- 完整 PTB dev 1700 句检查随后完成，耗时约 `164` 秒；MCTB 五域各 128 句仍只用于预测暴露统计。

#### 主要结果

完整检查结果：

- raw candidate keys: `275162`
- candidates with PTB rescues: `1346`
- loose passing candidates: `628`
- PTB dev chart spans / decoded spans: `555987 / 24843`
- PTB dev rescue / regression opportunity events: `2318 / 60968`
- MCTB 128x5 chart spans: dialogue `15687`, forum `37022`, law `50716`, literature `60557`, review `19423`

按 all-chart exposure 排名的候选大多没有 decoded-tree exposure。例如 `NX+NX -> NX` 在 MCTB chart 中出现 `3438` 次，但已解码树触发为 `0`。这类候选说明局部图表信号很多，但不会直接改变最终预测树。

按 decoded-tree exposure 看，仍然很窄：

| Candidate | PTB rescues / regressions / churn | MCTB chart / decoded exposure | 评价 |
| --- | --- | --- | --- |
| `ADJP -> LST`, rank&lt;=3, margin&lt;=1.0 | 2 / 0 / 2 | 181 / 4 | decoded 最高但标签语义怪，风险高 |
| `S -> VP`, rank&lt;=2, margin&lt;=1.0, len 11-20 | 1 / 0 / 0 | 95 / 3 | 最干净的微烟测线索 |
| `SINV -> S`, rank&lt;=3, margin&lt;=1.0 | 1 / 0 / 0 | 469 / 2 | 覆盖仍低，结构影响需谨慎 |
| `SBARQ -> FRAG`, rank&lt;=3, margin&lt;=0.5 | 1 / 0 / 0 | 348 / 2 | 可能是碎片化域特征，但证据弱 |
| `NX -> NP`, rank&lt;=3, margin&lt;=2.0, len 3-5 | 4 / 0 / 1 | 227 / 2 | PTB rescue 多一些，但已有 NX 类风险既往 |

#### 判断

直接 post-decode chart-label replacement 不值得作为主路线推进。原因有三点：

1. 宽松通过候选多，但最终解码树触发很低；最高只有 `4` 次，仍然接近前两轮窄规则的覆盖瓶颈。
2. 高 chart exposure 候选多数不在 decoded tree 上，后处理无法利用这些信号。
3. 部分 decoded 候选的标签语义不稳或属于既往高风险类型，例如 `ADJP -> LST` 和部分 `NX` 类变化。

本轮最有价值的发现是：模型 chart 中确实存在较宽的 gold-alternative signal，但如果要利用它，需要在解码前或解码过程中改变局部标签分数，而不是等 best tree 产出后再替换少数标签。

#### 下一步

保留诊断脚本和完整结果。下一步只做一个严格 feature-flagged 的解码前标签 bias 微烟测，而不是扩大后处理规则包。

推荐第一候选是 `S -> VP`, rank&lt;=2, margin&lt;=1.0, len 11-20。它不是覆盖最高的 chart 候选，但 PTB dev 风险最干净：`1` rescue、`0` regression、`0` churn，并且 MCTB 128x5 有 `3` 次 decoded exposure。若 feature-off parity、PTB dev 安全门槛或 MCTB prediction exposure 不成立，应停止当前后处理/图表标签线，转向更底层的训练目标或解码结构候选。

#### 降级记录

- 降级：直接 chart-level label alternative mining -&gt; post-decode 小规则包。
- 保留：`screen_chart_label_alternatives.py` 作为候选挖掘和失败分析工具。
- 保留：两规则小包作为窄安全补丁证据，但不全量升级。
- 暂不选择：`ADJP -> LST`，虽然 MCTB decoded exposure 最高，但标签语义风险和 churn 都比 `S -> VP` 更大。

### 阶段 6：解码前标签偏置微烟测

#### 结论

本轮完成了 `S -> VP` 解码前标签 bias 微烟测，并把当前 post-decode / chart-label 校准线整体降级为诊断资产，而不是继续扩大实验。

安全 bias 档位 `0.5 / 1.0 / 1.05` 在 PTB dev 128 句上完全不改变评测签名，labeled F1 都保持 `93.7448`；但它们对 MCTB 128x5 的最终预测影响极小，`1.0 / 1.05` 总共只改变 `2` 棵树。强 bias `2.0` 虽然能改变更多树，但 PTB dev labeled F1 从 `93.7448` 降到 `93.4072`，出现 `13` 个句级 loss。

因此，这条线的底层问题已经很清楚：chart 里有信号，但当前单标签局部 bias 不是足够强、足够稳的桥。它要么几乎不影响最终解码，要么一放大就开始破坏 PTB dev 安全门槛。

#### 本轮目标

1. 验证第 018 轮筛出的 `S -> VP` 候选是否能通过解码前分数偏置进入最终树。
2. 先检查 feature-off / `bias=0` 是否保持评测签名不变。
3. 在不使用 MCTB gold 的前提下，只观察 MCTB 预测树是否发生足够多、足够可解释的变化。

#### 实现

新增诊断脚本 `probe_predecode_label_bias.py`。脚本读取同一个 strong Berkeley+BERT checkpoint，复制 chart label score，在满足固定候选条件的 span 上给目标标签加一个小 bias，然后调用现有解码函数重新解码。

本轮候选固定为：

- from/to label: `S -> VP`
- rank limit: `2`
- max margin: `1.0`
- length bucket: `11-20`
- bias values: `0, 0.5, 1.0, 1.05, 2.0`

脚本现在把安全检查拆成两层：

1. signature parity：评测用 span/label 签名与 F1 是否不变。
2. tree-string parity：树字符串是否逐字完全一样。

这次结果显示 signature parity 通过，但 tree-string parity 失败。这更像重新解码后的字符串序列化差异，而不是评测层面的结构变化；所以最终判断以 signature parity、PTB dev F1 和非零 bias 的真实 changed count 为主。

#### 实验过程

- 本地语法检查通过。
- 远端 2 句 smoke 通过，证明 `bias=0`、非零 bias 和 no-gold MCTB 模式都能正常运行。
- 远端正式微烟测使用 PTB dev 128 句和 MCTB 五域各 128 句。
- MCTB 五域显式以 no-gold 模式运行，只统计预测变化，不用目标域标签做筛选。

#### 主要结果

PTB dev：

| bias | eligible | changed sentences | labeled F1 | delta labeled F1 | wins/losses |
| --- | --- | --- | --- | --- | --- |
| 0.0 | 39 | 0 | 93.7448 | 0.0000 | 0/0 |
| 0.5 | 39 | 0 | 93.7448 | 0.0000 | 0/0 |
| 1.0 | 39 | 0 | 93.7448 | 0.0000 | 0/0 |
| 1.05 | 39 | 0 | 93.7448 | 0.0000 | 0/0 |
| 2.0 | 39 | 13 | 93.4072 | \-0.3376 | 0/13 |

MCTB no-gold 预测变化：

| domain | changed at 1.0 / 1.05 | changed at 2.0 |
| --- | --- | --- |
| dialogue | 1 / 1 | 5 |
| forum | 0 / 0 | 6 |
| law | 0 / 0 | 9 |
| literature | 0 / 0 | 4 |
| review | 1 / 1 | 9 |

Feature-off 检查：

- PTB dev 和五个 MCTB no-gold 子集的 signature parity 均通过。
- tree-string exact parity 均失败，因此不能把它写成“字面输出完全不变”。
- 但 `bias=0` 的 changed sentence 为 `0`，PTB dev F1 也完全一致，说明评测签名层面没有变化。

#### 判断

当前候选不值得升级到全量 MCTB 或更大 3090 任务：

1. 安全档位几乎不改变最终树，跨域预测暴露仍停在个位数。
2. 强档位确实能扩大变化，但立刻带来 PTB dev 句级损伤和 F1 下降。
3. 这说明“局部单标签加分”无法稳定解决图表信号进不了最终树的问题。

这一轮也解释了为什么前几轮会反复卡在低覆盖上：如果只在最终树附近做局部标签修补，系统无法进入真正有信息的 chart 候选区域；如果强行推它进入，解码会开始破坏原来正确的 PTB 结构。

#### 下一步

停止继续做本地标签规则、chart-label 后处理、单标签 predecode bias 的小变体。下一步应先做一版更深层的机制复盘，把失败分成至少三类：

1. 候选存在但不能进入最终树。
2. 能进入最终树但覆盖太小。
3. 强行扩大后破坏 split / rule / label 组合稳定性。

然后再从更底层机制里选新候选，例如训练侧结构一致性目标、真正的 n-best / forest-level 重排，或更全局的 span-label composition 约束。新候选必须先说明它如何避免重复当前的“局部安全但无效、全局有效但伤分”的二分困境。

#### 降级记录

- 降级：`S -> VP` predecode single-label bias bridge。
- 降级：继续扩大 post-decode / chart-label local rule sweep。
- 保留：`probe_predecode_label_bias.py` 作为后续检测“局部 bias 是否真的影响最终解码”的工具。
- 保留：第 016 轮两规则小包作为窄安全补丁证据，但不全量升级。

### 阶段 7：机制失败复盘与下一候选选择

#### 结论

第 014 到 019 轮已经足够说明一件事：当前瓶颈不是“找不到局部正确标签”，而是局部候选信号很难变成一棵更好的完整解析树。继续做局部标签规则、margin 阈值或单标签 bias 调参，预期收益很低。

本轮把下一步收窄为三个真正不同的机制方向，并选择“整树候选生成 + 安全重排诊断”作为下一轮小验证。它的目标不是马上追全量指标，而是先回答一个更底层的问题：强 Berkeley+BERT 的图表里，是否存在可被稳定生成出来的近邻完整树；如果有，PTB-only 的全局结构特征能不能在不伤 PTB dev 的前提下选中更好的树。

#### 依赖证据

1. 已确认的比较边界：PTB-only 训练，MCTB_en test-only，对照采用 Berkeley+BERT。
2. 第 018 轮：完整 PTB dev 有 `2318` 个 top-3 金标标签备选事件，说明图表里有信号；但 MCTB 128x5 已解码树最高触发只有 `4` 次，说明 best-tree 后处理吃不到这些信号。
3. 第 019 轮：`S -> VP` 解码前 bias 在安全档几乎不改变最终树；强档改变更多树但让 PTB dev 128 句 labeled F1 从 `93.7448` 降到 `93.4072`。
4. 父线证据：tree-rule decode、soft rule decode、confidence-gated rule decode 都证明粗粒度 PTB 规则先验容易造成 split/rule churn；相反，窄标签小包虽然安全，但跨域触发只有 `2-3` 次。

#### 更底层的失败分解

| 层次 | 现在看到的现象 | 含义 |
| --- | --- | --- |
| 图表候选层 | gold alternative 很多，局部 top-k 里并不缺信号 | 问题不在局部标签空间完全没候选 |
| 解码选择层 | 大量 chart exposure 不进入最终树 | 局部分数改变不等于整树选择改变 |
| 全局一致性层 | 规则/组合先验一开大就伤 PTB 和 MCTB | 粗全局约束会引入过多结构 churn |
| 迁移安全层 | 窄规则安全但覆盖极低，强 bias 有伤害 | 下一步必须同时检查“能生成完整候选树”和“能安全选择” |

这和用户提醒的“不要停在表面指标”一致：最终 F1 只是结果，更决定性的指标是候选是否能进入整树搜索、整树近邻是否足够多、选择器是否能区分结构价值与结构噪声。

#### 文献和既有路线复核

本轮复用了已有文献图谱，并补读/复核了与当前瓶颈最相关的几类工作：

- Fried, Kitaev, and Klein 2019 指出跨领域泛化仍受结构化输出预测影响，强表示并不能自动解决整树泛化问题。
- Fried, Stern, and Klein 2017 说明 constituency parsing 的 reranking / model combination 有实际价值，但重排收益可能来自候选生成和模型组合共同作用，不能只做事后打分。
- Cui, Yang, and Zhang 2021 的非局部特征工作支持“span 局部预测需要与更大结构模式一致”这一判断。
- Zhang, Zhou, and Li 2020 的 neural CRF parser 支持全局树分布/二阶段 bracketing-labeling 是可行替代，而不是只能调独立 span label。
- Wang et al. 2022 的 order-sensitive decoding 支持在树组合时引入更细粒度的组合分数，但我们已有粗规则先验失败，因此不能直接重启规则分数 sweep。

本轮输出只是优化候选选择，不声称 paper-ready novelty。文献作用是约束机制方向：下一步要处理整树候选和全局选择，而不是继续堆局部标签规则。

#### 候选方向

| 候选 | 机制层 | 支持证据 | 最大风险 | 判断 |
| --- | --- | --- | --- | --- |
| A. 整树候选生成 + 安全重排诊断 | 解码后但以完整树为单位 | 直接回答 解析图信号能否进入完整树；不需要重训；可复用现有 chart 导出和审计脚本 | 如果候选生成只产生噪声，可能仍是 no-op | 选择 |
| B. 训练侧非局部一致性目标 | 训练目标 / 表示学习 | 文献直接支持，能从源头改变 scorer，使候选更可能进入最终树 | 需要重训，验证成本更高；之前结构一致性线有负诊断，必须重新设计 | 第二优先级 |
| C. 结构价值硬负例选择 | 训练目标 / 强负例 | 早期 uncapped contrastive 有强信号，用户也关心更底层指标 | 多轮 cap/quality gate 已失败，容易再次陷入局部 loss 工程 | 保留但不先选 |

被明确拒绝的方向：

- 继续 post-decode 小标签规则：覆盖太低。
- 继续单标签 pre-decode bias：安全档 no-op，强档伤 PTB dev。
- 直接重启 CKY rule bonus 或 threshold sweep：已有强 checkpoint 上系统性降级。

#### 选中候选 A 的原因

候选 A 现在最划算，因为它能用最小计算量回答下一步训练是否值得做：

1. 如果整树候选池里根本没有可恢复的 PTB dev 改善空间，训练侧大改也可能缺少明确目标。
2. 如果候选池有 oracle 空间但简单选择器选不中，下一步应转向非局部一致性或 learned reranker。
3. 如果候选池和选择器都在 PTB dev 小样本上安全，再看 MCTB no-gold prediction exposure，才有资格进入更大验证。

#### 下一轮小验证设计

目标：构造一个不改变训练、不使用 MCTB gold 的整树候选诊断。

最小实现：

1. 对同一 strong checkpoint 导出 chart 分数和标准解码树。
2. 用受控扰动生成每句 `3-8` 棵完整候选树，例如 label top-k 局部替换、轻量 score temperature、以及已有 soft-rule/top-k 工具产生的近邻树；每个候选必须是完整合法树。
3. 对 PTB dev 计算三种量：
   - candidate diversity：每句能生成几棵与标准树不同的完整树；
   - oracle delta：候选池里是否存在比标准树更接近 gold 的树；
   - safe selector delta：只用 PTB-derived 全局特征选择时是否不伤 labeled/unlabeled F1。
4. 对 MCTB 只看 no-gold exposure：候选池是否真正产生跨域完整树变化，不用 gold 调参。

成功标准：

- feature-off 完全保持标准解码；
- PTB dev 小样本 oracle delta 为正，且候选树变化不是主要来自 split 爆炸；
- safe selector 至少不明显伤 PTB dev；
- MCTB 五域各 128 句能产生非零且分散的完整树候选变化。

降级标准：

- 候选生成退化成重复标准树；
- oracle 有空间但所有简单结构特征都只会伤 PTB dev；
- MCTB 仍只有个位数完整树变化；
- 候选变化主要是 boundary/split churn，而不是标签或组合质量改善。

#### 下一步

把候选 A 提交为新的优化候选，并在当前工作区补一个独立诊断脚本。该脚本第一阶段只做候选池和 oracle/safety 统计，不进入全量 MCTB 评测，也不启动新训练。
