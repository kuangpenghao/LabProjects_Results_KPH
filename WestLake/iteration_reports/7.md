# 第007主版本：整树候选池与 选择器审计

## 结论

本主版本合并整树候选池、selector 修复、完整 PTB dev 安全验证、MCTB 子集验证和 MCTB 1000x5 放大判断。结论是：整树候选池值得保留，但当前无标签 selector 不足以稳定放大，阈值线已降级。

## 关键证据

- PTB dev 128 句中，候选 oracle 可把 labeled F1 从 93.7448 提到约 94.3008，增量约 +0.5560；oracle rescue/regression 为 10/0。
- 原始 margin selector 在同一批 PTB dev 128 句上为 -0.1112，selector rescue/regression 为 1/3。
- 加入 target-empty margin 和 minimum-span gate 后，小样本退化可被压到 0，但救回也很少。
- 完整 PTB dev 和 MCTB 固定评分没有证明该 selector 可以稳定迁移。
- MCTB 五域 1000 句放大后，阈值线没有稳住，因此不能作为主路线继续扩大。

## 深层判断

候选树空间确实有潜力，问题不在“没有更好的树”，而在“不用目标域标签时如何选对树”。这比继续调单一阈值更底层：selector 必须理解候选对全局结构的影响。

## 路线变化

保留整树候选池和 oracle 分析作为上限证据，但停止扩大当前阈值 selector。下一步转向全局结构风险和训练侧结构一致性。

## 完整阶段记录

本节保留本主版本内的连续实现、验证、问题定位和判断依据。后续同一方向的补充验证继续写入本主版本，只有研究问题、候选机制、比较合同或结论发生实质变化时才新开主版本。

### 阶段 1：整树候选池与安全筛选诊断

#### 结论

这轮结果把当前方向分成了两部分：

1. 整树候选池值得保留。PTB dev 128 句里，候选 oracle 可以把 labeled F1 从 `93.7448` 提到约 `94.3008`，增量约 `+0.5560`；其中 `10` 句可被候选救回，`0` 句被 oracle 伤害。
2. 当前无标签选择器不能放大。margin selector 在同一批 PTB dev 128 句上是 `-0.1112`，只有 `1` 次 rescue，却有 `3` 次 regression。
3. MCTB 五域无标签曝光不是空的：dialogue/forum/law/literature/review 分别有 `18/27/20/26/22` 句产生完整候选树变化。

因此，当前决策不是直接扩大 MCTB 评分，也不是放弃整树候选方向，而是继续同一条线做 selector repair：先修复“不用目标域标签也能选对候选”的问题。

#### 为什么做这一轮

第 018 和第 019 轮都显示了同一个底层瓶颈：图表内部存在可用标签信号，但它很少变成最终解析树。直接做后处理标签替换覆盖太窄，解码前单标签 bias 又在安全档近似无效、强档伤 PTB。

所以这一轮改问一个更深的问题：如果我们不只改最终树上的一个标签，而是从图表扰动重新解出完整树，正确结构是否已经藏在附近的完整候选树里？

#### 实验设计

- 固定强 Berkeley+BERT checkpoint，不重训。
- 每个非控制候选只做一次受控图表标签扰动，然后重新解码为完整树。
- PTB dev 使用 gold 只计算 oracle 上限和安全上限；不用 gold 的 selector 单独统计。
- MCTB 五域只做无标签完整树曝光统计，不用目标域标签做筛选、调参或早停。
- feature-off / identity gate 检查同一 checkpoint 下控制解码是否保持评测签名一致。

| PTB dev | yes | 128 | 31 | 31 | 93.7448 | +0.5560 | +0.5560 | \-0.1112 |
| MCTB dialogue | no | 128 | 18 | 18 | n/a | n/a | n/a | n/a |
| MCTB forum | no | 128 | 27 | 27 | n/a | n/a | n/a | n/a |
| MCTB law | no | 128 | 20 | 19 | n/a | n/a | n/a | n/a |

| MCTB review | no | 128 | 22 | 22 | n/a | n/a | n/a | n/a |

PTB dev oracle 的 wins/ties/losses 是 `10 / 118 / 0`，而 selector 是 `1 / 124 / 3`。这说明候选池本身有上限，但当前选择规则把好候选和坏候选混在一起了。

identity gate 的评测签名 mismatch 为 `0`。树字符串 mismatch 在 PTB 与 MCTB 均有少量出现，这是 chart-copy 重新序列化造成的字符串差异，不是评测签名差异。

#### 机制判断

这轮比前两轮更有价值，因为它证明了“图表信号无法进入最终树”不是绝对阻塞。通过完整树候选重解码，PTB dev 已经出现 `10` 个 oracle 可救句子，MCTB 五域也都有非零完整树变化。

但 selector 失败也同样重要：单纯按候选 margin 或局部分数选择，不能可靠区分“修正了标签/局部组合”的候选和“破坏了全局树”的候选。下一步 selector 必须引入更底层的关系特征，而不是继续调一个单阈值。

优先检查的 selector 特征：

- 候选与控制树的局部关系：是否只改标签、是否改 split、是否改变父子规则签名。
- 原始标签与候选标签的相对置信度，而不是只看单个 margin。
- 候选变化发生在整树的哪个结构位置：根附近、长 span、短 span、单子树替换的风险不同。
- 每句风险预算：一棵树最多接受多少处候选变化。
- 候选是否重复触发已知失败模式，例如 NX、SBAR+S+VP 这类既往造成目标域退化的模式。

#### 路线选择

| 候选动作 | 决策 | 理由 |
| --- | --- | --- |
| 直接扩大当前 selector 到 MCTB 评分 | 拒绝 | PTB dev selector 已经是负增益，且 regressions 多于 rescues。 |
| 放弃整树候选池 | 拒绝 | PTB oracle `+0.5560` 且 `10/0` rescue/regression，MCTB 五域也有完整树曝光。 |
| 继续同线修复 selector | 选择 | 候选池存在，瓶颈集中在不用 gold 的选择规则。 |
| 切到训练侧一致性目标 | 保留为第二优先级 | 如果修复 selector 后仍无法安全选择候选，再转向训练侧。 |

#### 下一步

1. 检查 PTB dev 128 中 选择器的 `1` 个 rescue 和 `3` 个 regression，定位选择器具体错在什么结构关系上。
2. 加一个更严格的无标签 选择器，先用候选关系特征和风险预算约束选择。
3. 重跑 PTB dev 128 selector-repair smoke；只有 selector 非负且 regression 不超过 rescue，才考虑 MCTB 评分。

#### 当前降级项

当前 margin selector 明确降级，不能作为跨域评分入口。未来若要重新打开它，至少需要证明 PTB dev 小样本非负、regressions 可控，并且不依赖 MCTB gold。

### 阶段 2：整树候选选择器修复小样本验证

#### 结论

这轮把第 021 轮失败的 selector 做了一个小而明确的修复：原始无标签 margin selector 在 PTB dev 128 上是负的，`1` 次救回但 `3` 次退化；加入 target-empty margin 后退化降到 `2` 次但仍为负；再加入 minimum-span gate 后，退化降到 `0` 次，同时保留 `1` 次救回。

因此当前判断是：selector repair 可以继续，但还不能当成最终跨域机制。它只在 128 句上证明了“短 span 噪声过滤”有效；下一步必须先用完整 PTB dev 验证安全性，再决定是否进入 MCTB 评分。

#### 为什么做这一轮

第 021 轮已经证明完整树候选池本身有潜力：oracle 在 PTB128 上是 `+0.5560` labeled-F1 points，且没有 oracle loss。但不用 gold 的 margin selector 是 `-0.1112` points，说明瓶颈不是“候选不存在”，而是“不知道什么时候该选候选”。

上一轮最直接的失败线索是：低 target-label margin 并不等于安全。一个候选即使接近当前非空标签第一名，也可能只是把很短的局部短语改成另一个常见标签，最终伤害整树 labeled F1。

#### 修复设计

这轮只改 selector 诊断脚本，不改主解析器，也不重训：

1. 记录 selector win / loss / tie 样例，分开看救回和退化。
2. 加 target-empty margin 下限，要求候选非空标签相对 empty / no-constituent 有足够优势。
3. 加 minimum-span gate，先排除最容易产生局部标签噪声的超短 span。

整个验证仍然只使用 PTB dev gold 来做安全审计；MCTB 不参与筛选、调参或早停。

#### 关键结果

| 设置 | selector delta | selector rescues | selector regressions | wins / losses / ties | 判断 |
| --- | --- | --- | --- | --- | --- |
| 原始 margin selector | `-0.1112` | 1 | 3 | `1 / 3 / 124` | 失败 |
| target-empty margin &gt;= 0.15 | `-0.0556` | 1 | 2 | `1 / 2 / 125` | 仍失败 |
| target-empty margin &gt;= 0.15, min span &gt;= 3 | `+0.0556` | 1 | 0 | `1 / 0 / 127` | 通过小样本门 |

通过设置下，oracle 仍保持 `10` 次救回、`0` 次退化，说明 candidate pool 没有被修复过程破坏；selector 只是更保守地从候选池里取了一小部分。

#### 失败机制解释

target-empty gate 能去掉一个长 span 退化，但剩下两个退化都有同一个更深特征：都是 length-2 的短 span relabel。

这说明这类 selector 不能只看“目标标签和当前标签的局部分差”。在很短短语上，局部标签替换经常只是标签歧义噪声，并不代表完整树变得更好。相反，唯一保留的救回发生在 length-17 的较大结构上，更像是候选树修复了一个真正的上层组合错误。

所以当前第一性原理判断是：整树候选 selector 应该优先保护短 span 的局部稳定性，把选择预算留给能改变高层组合错误的候选。这个判断比“继续调 margin 阈值”更有解释力。

#### 路线选择

| 候选动作 | 决策 | 理由 |
| --- | --- | --- |
| 直接进入 MCTB 评分 | 暂缓 | PTB128 通过但样本太小，且只有 `1` 个实际救回。 |
| 放弃 whole-tree selector | 拒绝 | 修复后已经把 `3` 个退化压到 `0` 个，说明失败不是不可修复。 |
| 扩大到完整 PTB dev 安全验证 | 选择 | 这是判断 selector 是否真正安全的最低成本下一步。 |
| 立刻设计更复杂 selector | 暂缓 | 先看固定双门在 1700 句上是否仍稳，再决定是否需要更复杂特征。 |

#### 下一步

1. 用固定设置 `target-empty margin >= 0.15`、`min span >= 3` 跑完整 PTB dev 安全验证。
2. 如果完整 PTB dev 仍非负且退化可控，再做 MCTB no-gold 曝光检查和一次固定-selector MCTB 评分。
3. 如果完整 PTB dev 失败，降级当前双门 选择器，并转向更深的 selector 特征，例如高层结构位置、split 保持、父子规则签名变化和每句风险预算。

#### 当前降级项

原始 margin selector 明确降级。target-empty-only selector 也降级：它能减少退化，但仍没有过 PTB128 安全门。

### 阶段 3：整树选择器 全量安全验证与跨域降级

#### 结论

当前 target-empty + minimum-span 双筛选门不能作为跨领域可放大的 selector。它在完整 PTB dev 上只是微弱正向，MCTB 128x5 固定评分转为负值；因此本轮将当前 selector 正式降级，不继续扩大同一规则。

更有价值的发现是：整树候选池本身仍然有跨域潜力。MCTB 五域的 oracle / safe-oracle 增益全为正，说明“好候选存在”，但现有无标签选择规则无法可靠地区分救回和退化。

#### 目标

本轮回答一个具体问题：第 022 轮在 PTB128 上通过的双筛选门，是否能在更完整的开发集和目标域小样本上保持安全。

成功标准：

1. 完整 PTB dev selector 至少非负，并且救回/退化比例不能接近随机。
2. MCTB 128x5 固定评分不能出现多数域下降。
3. 如果 selector 失败，必须区分是候选池没有潜力，还是无标签选择规则选错。

#### 实验设置

- 方法：整树候选生成 + 无标签 selector。
- 固定 selector 门槛：`target_empty_margin >= 0.15`，`span_len >= 3`。
- PTB dev：1700 句完整开发集。
- MCTB：dialogue / forum / law / literature / review 各 128 句。
- 约束：不使用 MCTB 标签做训练、筛选或调参；MCTB 标签仅用于本轮固定方案评分与降级判断。

| PTB dev 1700 | 93.9136 | +0.3158 | +0.3158 | +0.0043 | 13 / 12 / 1675 | 73 / 0 / 1627 |
| MCTB dialogue 128 | 83.1306 | +0.5288 | +0.5288 | \-0.3173 | 1 / 4 / 123 | 5 / 0 / 123 |

MCTB selector macro delta 为 `-0.0765`，safe-oracle macro delta 为 `+0.5133`。这说明跨域候选空间不是空的；失败集中在 selector。

#### 诊断

1. **PTB dev 通过得太勉强。** 完整开发集上 selector 只有 `13` 个救回和 `12` 个退化，最终 LF1 只涨 `+0.0043`。这不是稳定机制，更像近似打平。
2. **MCTB 负面结果集中在选择错误。** Dialogue、forum、law 三域下降，且这些下降发生在 oracle 全正的情况下；因此不能把失败归因于没有可用候选。
3. **跨度和空标签 margin 只是浅层安全门。** 第 022 轮中它们能挡住明显短跨度错误，但完整 PTB dev 和 MCTB 中救回与退化都覆盖 3-20 左右的中等跨度，单靠这两个门槛无法表达“结构是否真的更可信”。
4. **当前 选择器没有域不变依据。** 目标域里大量候选是边界不变、标签变化的局部替换；它需要更深的结构依据，例如候选是否保持父子关系、是否减少不稳定组合、是否符合句子级风险预算，而不是只看局部分数关系。

#### 决策

| 选项 | 决策 | 理由 |
| --- | --- | --- |
| 继续放大当前双筛选门 selector | 拒绝 | MCTB 128x5 平均为负，且 3/5 域下降。 |
| 放弃整树候选池 | 拒绝 | Oracle / safe-oracle 在 PTB dev 与五个 MCTB 域均为正，候选池仍有研究价值。 |
| 修复 selector 特征后再评估 | 选择 | 当前证据把瓶颈定位在无标签选择规则，下一步应分析 target-domain win/loss 并设计更域不变的 selector。 |
| 立即转训练侧一致性目标 | 暂缓 | 训练侧目标更重；若 selector 分析显示无标签选择无法稳定，才把它作为下一主线。 |

#### 下一步

下一步不再调同一个 `target_empty_margin` / `span_len` 门槛。应该先做 selector 失败分析，重点比较 MCTB 中 selector wins 与 losses 的更底层差异：

- 标签变化是否跨越高风险父子组合或复合标签；
- 候选是否保持边界但破坏局部结构关系；
- 句子中候选数量、分数分布和风险预算是否能解释错误选择；
- 是否需要从单候选贪心选择改成句子级“最多一次、需多证据同意”的保守选择。

如果这些特征不能把 wins/losses 明确分开，就应停止无标签后处理 选择器，转向训练侧结构一致性或更真实的 n-best/forest reranking。

### 阶段 4：空结构风险门控 选择器复核

#### 结论

本轮把第 023 轮失败的整树选择器 从“跨域负向”修成了“小幅正向但仍待验证”的候选。只收紧分数差门槛时，PTB dev 从 `+0.0043` 到 `+0.0087`，MCTB 128x5 从 `-0.0765` 到几乎为零；再加入空结构倾向上限后，PTB dev 达到 `+0.0130`，MCTB 128x5 宏平均达到 `+0.0353`，五个目标域均不再负向。

这说明上轮失败不是候选池无效，而是无标签 selector 需要避开“模型过强地想把结构删空”的位置。当前信号更干净，但收益仍很小，不能当作最终算法提升。

#### 本轮目的

第 023 轮已经证明两件事：

1. 整树候选池有价值：MCTB 五域 safe-oracle 都为正，宏平均约 `+0.5133`。
2. 当时的无标签 selector 不可靠：完整 PTB dev 只有 `+0.0043`，MCTB 128x5 为 `-0.0765`，并且 3/5 个域下降。

所以本轮不继续盲目放大，而是先问一个更底层的问题：选错的候选是否有可分离的风险特征？如果没有，就应停止阈值微调，转向训练侧一致性或真正的 n-best / forest reranking。

#### 实现改动

本轮只做默认关闭的风险门控，不改训练、不改主解析器路径：

- 保留第 023 轮的 `target_empty_margin >= 0.15` 与 `span_len >= 3`。
- 将 score-margin 上限从 `0.25` 收紧到 `0.075`，检查是否只在模型接近犹豫时改树。
- 新增 `--selector-max-empty-margin`，当目标空结构倾向过强时拒绝该候选。
- 最终复核配置使用 `max_empty_margin = 1.3`。

这个改动的含义是：selector 不只看“能不能改”，还看“这次改动是不是建立在过强的删空倾向上”。后者更接近这轮残余错误的机制原因。

#### 实验设置

使用同一个完整 Berkeley+BERT 强 checkpoint 与同一套固定评分入口：

- PTB dev：1700 句完整开发集。
- MCTB 128x5：dialogue、forum、law、literature、review 每域 128 句。
- 训练数据约束：不使用 MCTB 标签训练、筛选、调参或早停。
- 对照方式：同 checkpoint 标准解码作为 control，selector 改动只在候选重排层发生。

| strict score-margin gate | `+0.0087` | `5/3/1692` | `+0.0003` | `5/5/630` | `0.0000 / -0.1260 / 0.0000 / +0.1276 / 0.0000` |
| strict + max-empty gate | `+0.0130` | `5/2/1693` | `+0.0353` | `5/2/633` | `0.0000 / 0.0000 / +0.0490 / +0.1276 / 0.0000` |

#### 机制判断

分数差门槛有用，但主要是止损：它把明显坏的跨域影响压掉，却没有创造稳定收益。真正更有解释力的信号是空结构倾向上限：残余错改里多次出现目标空结构 margin 很高的情况，说明 selector 容易在“模型强烈想删掉结构”的位置过度相信候选。

加入上限后，跨域五域不再出现负向域，这比单纯调 score-margin 更像一个机制性修复。不过当前覆盖仍低，整体增益只有 `+0.0353`，离可推广算法还差得比较远。

#### 当前评价

本轮候选从“降级”恢复为“可做一次更大验证的轻量候选”，但不能直接升级为主方案。

保留理由：

- PTB dev 与 MCTB 128x5 同时非负。
- MCTB 五域无负向域。
- 修复特征来自失败样例，而不是盲调阈值。

不升级理由：

- MCTB 宏平均增益只有 `+0.0353`。
- safe-oracle 仍有约 `+0.5133`，说明大部分候选空间还没有被可靠选择。
- 如果更大验证后覆盖仍低，这条线会变成过窄后处理补丁，而不是跨域解析机制改进。

#### 下一步

建议只追加一次更大范围 MCTB 验证。如果更大验证保持非负并有足够覆盖，再考虑把 max-empty selector 作为候选方案继续修；如果转为中性、负向，或仍只有极少触发，应停止 selector 阈值微调，转向更有广度的机制：

1. 训练侧结构一致性目标。
2. 真正的 n-best / forest reranking。
3. 能直接控制 split / rule-signature churn 的全局校准方法。

### 阶段 5：MCTB 1000x5 放大验证与选择器阈值线降级

#### 结论

第 024 轮的 max-empty selector 在更大 MCTB 验证上没有稳住。五域各 1000 句、总计 5000 句后，selector 宏平均只剩 `+0.0048` labeled-F1 points，literature 域转为 `-0.0241`，整体只有 `21` 胜、`19` 负、`4960` 平。

这不是值得继续阈值微调的信号。正确动作是停止当前 selector-threshold 路线，把整树候选池和 max-empty 风险特征保留下来，下一轮转向更全局、更有广度的选择机制。

#### 本轮目的

第 024 轮的小样本结果较干净：

- PTB dev：`+0.0130`。
- MCTB 128x5：宏平均 `+0.0353`。
- 五域无负向。

但该收益非常小，所以不能直接升级。本轮只回答一个问题：这个小正向是否能在更大目标域样本上保持非负并有足够覆盖？

#### 实验设置

设置保持第 024 轮最终 selector 不变：

- checkpoint：同一个完整 Berkeley+BERT 强 checkpoint。
- 数据：MCTB_en 五域，每域 1000 句。
- 训练：无训练，checkpoint 不变。
- selector：`target_empty_margin >= 0.15`、`target_empty_margin <= 1.3`、`span_len >= 3`、`target_margin <= 0.075`。
- 约束：不使用 MCTB 标签训练、筛选、调参或早停；MCTB 标签只用于这次固定方案的最终评分。

#### 结果

| 域 | selector delta | changed-candidate sentences | safe-label sentences | W/L/T |
| --- | --- | --- | --- | --- |
| dialogue | `+0.0268` | 155 | 155 | `3/1/996` |
| forum | `+0.0152` | 246 | 246 | `8/6/986` |
| law | `+0.0063` | 145 | 144 | `4/3/993` |
| literature | `-0.0241` | 233 | 231 | `3/6/991` |
| review | `0.0000` | 175 | 174 | `3/3/994` |
| macro / total | `+0.0048` | 954 | 950 | `21/19/4960` |

候选池本身仍有明显空间：五域 oracle delta 分别为 `+0.5943 / +0.5616 / +0.2412 / +0.4664 / +0.6909`，safe-oracle 也大致保持半个点量级。但当前无标签 selector 只能捕捉到极小一部分，且跨域稳定性不足。

#### 机制判断

第 024 轮的 max-empty gate 确实修掉了一部分“过强删空倾向”错误，但它不是足够完整的选择机制。放大后问题变得清楚：

1. 净胜数量太小：`21` 胜对 `19` 负，5000 句里几乎没有实质净效果。
2. 域间不稳：literature 从小样本非负变成负向，说明小样本五域无负不能作为可靠升级依据。
3. oracle 空间仍大：候选树里有好解，但 threshold selector 无法稳定挑中它们。

这意味着当前瓶颈不是再找一个更细阈值，而是需要一个更全局的选择原则。

#### 当前评价

降级对象：

- 当前 max-empty-gated threshold selector 作为性能路线降级。
- 继续在 128x5 或 1000x5 上微调 `target_margin`、`empty_margin`、`span_len` 阈值的路线降级。

保留对象：

- 整树候选池：oracle / safe-oracle 仍说明好候选存在。
- max-empty risk：它是有解释力的风险特征，可作为未来更全局 选择器的一个输入。
- 当前评测脚本和固定评分入口：可复用为下一轮 candidate 选择器的验证工具。

#### 下一步

下一轮不再继续 threshold selector 微调。建议进入 broader selector / mechanism selection，优先比较三个方向：

1. true n-best / forest reranking：让 selector 直接比较完整树而不是单一局部事件。
2. training-side structure consistency：训练阶段降低 split / rule-signature churn。
3. global churn-aware selector：显式惩罚会引入 split、rule-signature 或过强 empty-structure 风险的候选树。

选择标准应是能否利用现有候选池的 oracle 空间，同时在 PTB dev 和 MCTB 固定验证上避免 `21/19` 这种几乎抵消的结果。
