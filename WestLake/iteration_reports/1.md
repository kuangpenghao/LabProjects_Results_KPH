# 第001主版本：基线合同与比较边界

## 结论

本主版本确认了后续优化使用的正式比较合同：Yang et al. 2022 Table 3 的 Berkeley+BERT，PTB 训练、MCTB_en 五域测试，macro F1 = 87.20。

这不是完整本地复现完成的结论。严格本地复现仍受 PTB/goldtags 文件缺失限制，不能把 tiny 或本地工程诊断当作正式 Berkeley+BERT 对比结果。

## 合同内容

- 正式基线：Berkeley+BERT。
- 训练域：PTB。
- 测试域：MCTB_en dialogue、forum、law、literature、review。
- 五域 F1：86.30、87.04、92.06、86.26、84.34。
- 宏平均 F1：87.20。

## 对后续工作的约束

后续任何性能声称都必须明确区分三类结果：

1. 正式可比结果：使用同一训练/测试边界，并覆盖合同指标。
2. 工程诊断结果：用于确认链路、脚本或机制是否能运行。
3. 候选筛选结果：用于决定是否继续，不可直接声称跨域提升。

## 当前判断

baseline gate 已打开，可以继续算法优化；但如果未来要声称超过 Berkeley+BERT，必须回到正式比较合同，而不是引用 tiny 模型或不完整链路。

### 阶段 1：基线复现与比较合同

#### 1. 本轮结论

本轮完成的是“可继续优化的基线合同”，而不是完整论文级本地复现。

- 正式比较基线采用 Yang et al. 2022 Table 3 的 Berkeley+BERT，设定为 PTB 训练、MCTB_en 五域测试。
- 本地 Berkeley parser tiny 链路已经跑通，覆盖训练、dev 评测、模型保存、模型加载、test 评测和 EVALB。
- strict Berkeley+BERT 本地复现暂时不能声称完成，因为公开仓库和本机都没有找到 `02-21.goldtags`、`22.goldtags` 这类 strict 命令需要的 PTB/goldtags 文件。
- 后续优化必须把 tiny 结果当工程诊断，把论文 Table 3 当正式比较参照。

#### 2. 本轮目标和上下文

项目目标是提升 constituency parser 的跨领域表现，训练必须来自 PTB，测试必须来自论文使用的 MCTB_en 跨领域数据集。

本轮要先回答三个基础问题：

1. 应该把哪个结果作为正式比较基线。
2. 本地 Berkeley parser 工程链路是否能跑通。
3. 本地是否已经具备 strict Berkeley+BERT 论文级复现条件。

本轮约束是：不能把 tiny 工程分数冒充正式跨领域 F1，也不能在缺少 PTB/goldtags 的情况下声称 strict Berkeley+BERT 已完整复现。

#### 3. 思路进展与依据

基线选择依据来自 Yang et al. 2022 的实验设置：该论文正是本项目的问题来源，报告了 Berkeley+BERT 在 MCTB_en 五个领域上的 F1。由于本地 strict 复现受 PTB/goldtags 文件缺失限制，最稳妥的做法是把论文表格结果固化为正式比较合同，同时把本地代码链路作为后续算法实现和诊断的工程底座。

正式基线保留分领域指标，而不是只保留一个平均值：

| 领域 | Berkeley+BERT F1 |
| --- | --- |
| dialogue | 86.30 |
| forum | 87.04 |
| law | 92.06 |
| literature | 86.26 |
| review | 84.34 |
| macro | 87.20 |

这也说明当前距离用户目标 95+ 仍有明显差距，尤其 dialogue、forum、literature、review 都低于 87.1。

#### 4. 具体算法与工程实现

本轮没有提出新算法，重点是恢复可用基线系统和比较合同。

主要工程内容：

- 恢复 `self-attentive-parser` 的 `acl2019` 版本作为 Berkeley parser 代码底座。
- 建立隔离 Python 环境，固定 Python 3.9、PyTorch 1.13.1+cu117、NumPy 1.26.4。
- 编译并验证 EVALB。
- 修复旧代码在现代 Python/PyTorch 下的兼容问题。
- 完成 tiny train/test，验证模型保存、加载和评测链路。
- 验证 BERT 权重入口可加载，确认 BERT 本身不是当前主要阻塞。
- 写入正式指标合同 JSON，后续实验必须按五域 F1 + macro F1 报告。

#### 5. 中间问题与解决方法

| 问题 | 影响 | 解决方法 | 结果 |
| --- | --- | --- | --- |
| 旧代码含 `cuda(async=True)` | Python 3.9 无法解析 | 改为 `cuda(non_blocking=True)` | 主入口可启动 |
| 旧 PyTorch `uint8` mask 语义变化 | 训练中张量长度不匹配 | 改成 bool mask | tiny train 通过 |
| NumPy 2.x 与旧 PyTorch/扩展不兼容 | 训练入口失败 | 隔离环境内固定 NumPy 1.26.4 | 环境恢复 |
| strict BERT 命令缺少 goldtags | 无法合法完整复现论文训练 | 降级为论文 Table 3 正式基线 + 本地 strict repair-pending | 基线门打开 |
| 基线状态同步出现最后一步异常 | 已写记录没有立刻成为当前入口状态 | 手动同步已确认基线引用 | 当前状态为 confirmed |

#### 6. 实验设计

本轮实验分两层：

1. 正式比较合同：采用论文 Table 3 Berkeley+BERT 五域结果，不重新解释为本地实验。
2. 本地工程诊断：用仓库自带 `.clean` 数据做 tiny train/test，验证代码链路能用于后续改算法。

tiny 诊断不使用 MCTB_en 作为训练数据，也不承担正式跨领域比较功能。

#### 7. 实验结果

本地 tiny 诊断结果：

- 训练样例：947，来自 `data/02-21.10way.clean`。
- dev 样例：48，来自 `data/22.auto.clean`。
- tiny dev：Recall 1.82 / Precision 0.76 / F1 1.07 / CompleteMatch 0.00。
- test 样例：2416，来自 `data/23.auto.clean`。
- tiny test：Recall 0.87 / Precision 0.34 / F1 0.49 / CompleteMatch 0.00。
- BERT 入口验证：`bert-base-uncased` 可加载，参数量 109,482,240。

这些低分是预期内的，因为它们来自 tiny 工程验证，不是完整 Berkeley+BERT 训练。

#### 8. 评价、经验与教训

本轮满足预期：基线比较合同已经可用，工程链路也可用于后续小规模诊断。

关键教训：

- “能跑 tiny”与“完成正式基线复现”必须严格区分。
- 后续任何优化结果都要保留五域指标，不能只汇报 macro。
- 没有合法 PTB/goldtags 时，不应该把时间无限投入 strict 复现重试；更高效的做法是先用可信合同推进算法方向，同时保留 strict 复现缺口。
